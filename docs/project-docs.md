# Aoroza 项目文档生成规范与逻辑说明

本文档总结了在 Aoroza 项目开发过程中，用于生成和管理项目文档（如实施计划、任务列表、功能演示与总结等）的逻辑、模板和相关规范。

## 1. 文档分类与作用

在协助开发本项目的过程中，主要使用了以下几类规范化文档来保障开发流程的清晰和可追溯：

### 1.1 实施计划 (Implementation Plan)
*   **用途**：在进行复杂的架构调整、重大功能新增或重构前，进行前期调研并给出详细的技术方案。
*   **核心逻辑**：无代码修改的纯技术调研 -> 列出涉及的组件与文件变动方案 -> 列出需要用户确认的开放问题 (Open Questions) -> 提交给用户 Review 并等待批准。

### 1.2 任务清单 (Task Tracker)
*   **用途**：作为执行阶段的动态 TODO list，用于跟踪复杂任务的进度。
*   **核心逻辑**：将大任务拆解为子任务（Component-level items），随着工作推进实时更新状态（`[ ]` 未完成, `[/]` 进行中, `[x]` 已完成）。

### 1.3 功能演示与总结 (Walkthrough / Changelog)
*   **用途**：在任务完成后，向用户总结已完成的修改、测试方式以及验证结果。
*   **核心逻辑**：作为不断累加的开发日志，后续相关的修改只需在原有文档中补充新的章节（如：文件树排序优化、项目重命名等），而不是每次都创建新文档。

### 1.4 项目审阅 (Project Review)
*   **用途**：用于盘点当前项目的文件结构、技术栈、核心模块以及未来的优化方向。
*   **核心逻辑**：对现有代码库的全面静态分析结果。

## 2. 常用文档编写模板

### 2.1 实施计划 (Implementation Plan) 模板

```markdown
# [目标描述]
简要描述问题背景、开发目标以及此次变更的预期效果。

## 需要用户审阅的内容 (User Review Required)
记录任何重大设计决策或可能产生破坏性变更 (Breaking changes) 的部分。
> [!IMPORTANT]
> 关键要求或必须知晓的信息。

## 待确认问题 (Open Questions)
任何可能影响方案设计的待确认细节。

## 预计修改内容 (Proposed Changes)
按组件或模块分组列出预计的文件变动：

### [模块名称]
*   **[MODIFY]** `src/components/Example.tsx`：说明修改细节...
*   **[NEW]** `src/utils/helper.ts`：说明新建文件目的...
*   **[DELETE]** `src/old_file.ts`：说明删除原因...

## 验证计划 (Verification Plan)
如何验证修改的正确性（包括自动化测试与手动测试步骤）。
```

### 2.2 功能总结 (Walkthrough) 模板

```markdown
## [序号]. [功能/修改名称]

### 功能描述
详细描述所做的更改及其技术实现细节：
1. **[核心改动 1]**：具体说明及原理。
2. **[核心改动 2]**：具体说明及原理。
3. **涉及修改的文件**：
   * `文件路径`：具体修改了什么。

### 验证结果
*   编译/构建是否成功？
*   功能测试结果如何（例如：排序功能测试，排重测试等）。
*   UI 交互层面的改善效果。
```

## 3. 文档编写规范与格式约束

1. **绝对路径与可点击链接**：
   在文档中提及具体的代码文件或行号时，强制使用 Markdown 格式的可点击链接（`[file basename](file:///absolute/path)`），以便于快速跳转查看。
2. **GitHub 风格的提示框 (Alerts)**：
   在关键信息处使用 GitHub 风格的 Alert 以提高可读性，如 `> [!NOTE]`, `> [!TIP]`, `> [!IMPORTANT]`, `> [!WARNING]` 等。
3. **差异代码块 (Diffs)**：
   在需要展示具体代码变动时，使用 Diff 格式的代码块直观地展示增减的内容。
4. **简洁明了**：
   优先使用无序列表、有条理的层次结构（H2, H3 等），避免出现过长的不换行段落，保证极佳的阅读体验（Scannability）。
